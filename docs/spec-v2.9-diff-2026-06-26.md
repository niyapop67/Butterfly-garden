## 2. アクセス制御：2段階の合言葉ゲートを新設（追加仕様）

v2.9には記載のなかった新しい仕様。1.11.1節の「Niya専用プレビューモード」（`?preview=`）とは
どちらも別物。3つの鍵は互いに独立しており、混同しないこと。

### ゲート1：サイト入場の合言葉（`SITE_PASSPHRASE`）

- **対象範囲**：トップページ・メッセージ投稿ページ・ガーデンページ・誕生日当日ページなど、
  サイトの全ページ
- **方式**：共通の合言葉（パスフレーズ）を1つ設定し、入力画面を最初の関所として設置
- **実装方法**：Next.js Middleware + Cookie（一度認証されたブラウザは再入力不要）
- **強度**：本格的なサーバーサイド認証ではなく、「URLが漏れても合言葉を知らない人は入れない」
  程度の軽量ゲート
- ゲート1を通った人がガーデンページで蝶をタップすると、**ニックネームのみ**表示される
  （メッセージ本文・ボイスはここでは見えない。下記ゲート2が別途必要）

### ゲート2：プライベート体験（本文・ボイス閲覧）の合言葉（`PRIVATE_EXPERIENCE_PASSPHRASE`）

- **対象範囲**：6章で設計された「MIKAプライベート体験」専用URL（おまかせ再生／ガーデン探索／
  名前一覧リストの3モード）のみ。誕生日当日ページ経由の「みんなの声を聴く」体験がこれに該当する
- ゲート1とは完全に別の文字列。ゲート1を通っていても、ゲート2を知らなければ本文・ボイスは
  一切見えない
- ここを通った人だけが、メッセージ本文とボイスを閲覧・再生できる
- **実装スコープの扱い（更新・2026-06-26）**：当初は仕組み（検証関数）のみ下準備する予定
  だったが、入力ページ（`/private/enter`）と検証API（`/api/private-gate`）も合わせて実装した。
  パス構成：
  - `/private/enter` — ゲート2の合言葉入力フォーム（ゲート1からは免除されており、
    ゲート1未通過でも到達できる）
  - `/api/private-gate` — 合言葉を検証し、`bg_private_gate` Cookieを発行
  - `/private/*`（その他の将来パス）— ゲート1・ゲート2の両方を要求。どちらかが
    欠けていると `/enter` または `/private/enter` にリダイレクトされる
  - 6章の3モード本体（おまかせ再生／ガーデン探索／名前一覧リスト）はこの `/private/*`
    配下に置く想定。まだページ自体は実装していない（今回のスコープ外のまま）
  - 動作確認済み：ゲート1のみでは`/private/*`に入れない、ゲート1+ゲート2両方で
    Middlewareを通過できる、の2点をcurlで検証済み

理由：身内（MIKA本人含む）向け運用を踏まえ、サイト全体への軽い関所（ゲート1）と、本文・
ボイスという最もプライベートな情報への関所（ゲート2）を分けることで、ガーデンページの
気軽な閲覧体験とプライベート体験の特別感を両立する。

## 3. ガーデンページでの公開範囲（v2.9 §2.3・§6 を補足）

- 蝶の数・人数カウント：投稿後すぐに反映（随時公開、v2.9の想定通り）
- ガーデンページで蝶をタップ：**ニックネームのみ**表示（ゲート1のみで見える）
- メッセージ本文・ボイスは、6章で設計された「MIKAプライベート体験」内でのみ閲覧可能
  （ゲート2の内側）

v2.9本体の6章（プライベート体験）の設計意図とは矛盾しない。ガーデンページ単体では
メッセージ本文を見せない、という運用上の補足。

Firestore保存 + Storage録音アップロードまで含めて、本番相当で一通り動く形を実装する。
- Firestoreへの参加者データ保存
- Firebase Storageへの録音（WebM）アップロード
- 取得・表示
- 上記の合言葉ゲート

## 5. 蝶アセットの未確定3種類への対応

v2.9 §1.8で確定している7種類のうち、画像素材が用意できているのは4色（Pink Heart / Tiffany Sky相当 / Aurora Dream相当 / 不明1種）のみ。Crystal White・Emerald Garden・Golden Sunshineの専用PNGは未着手。

- 実装は7種類の `type` を正式に受け付ける構造にする
- 画像未確定の3種は、暫定的に既存色を仮表示として割り当てる（`src/lib/butterflyAssets.ts` で一元管理）
- 最終PNGが用意できた時点で、`butterflyAssets.ts` のマッピングを変更するだけで反映される設計とする（コンポーネント側のコード変更は不要）

## 6. 付随対応：Next.jsのセキュリティパッチ更新

実装と直接関係はないが、`npm install` 時に `next@14.2.5` の既知の脆弱性警告が出たため、
パッチ版 `14.2.35` に更新した（2025年12月のReact Server Components関連CVE群への対応）。
ビルド・型チェックともに問題なし。今後 `package.json` を見て「なぜ14.2.35なのか」が
分かるように、ここに記録しておく。
